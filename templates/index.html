Herein lie the endpoints that can be used for testing responses using the TransactID library. These end points are top level and generic here. In your own instances you would want to have an identifier for each of your customers who are using this. E.g. "/customer_identifier/invoice-request" which you'll be using as the relivent endpoint when building out the URLs.

There are four endpoints (besides this one, hi!) that you can use for testing. The first of these is only for this testing and won't be used in the wild.

1.
Endpoint: /initial-invoice-request
Verb: GET
Get Response: Binary object
Verb: POST
POST URL Parameter: url=[your invoice-request endpoint]
POST Response: Str showing success or not sending to provided URL.
Notes: just send a GET request to this to receive back an invoice-request binary so that you can test parsing things. If you want to test your full flow with getting an invoice-request object at your correct endpoint use the POST as described and it will send the binary object to that URL.

2.
Endpoint: /invoice-request
Verb: POST
Payload: binary object
Response: binary serialized_payment_request
Notes: send a binary created using the create_invoice_request method to this endpoint and receive a payment request in return.

3.
Endpoint: /payment-request
Verb: POST
Payload: binary object
Response: binary serialized_payment
Notes: send a binary created using the create_payment_request method to this endpoint and receive a payment object in return.

4.
Endpoint: /payment
Verb: POST
Payload: binary object
Response: binary serialized_payment_ack
Notes: send a binary created using the create_payment method to this endpoint and receive a payment_ack in return.
